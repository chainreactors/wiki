

## 背景

IoM目前只有三个核心开发者。我们有先进的设计和宏伟的计划，但受限于时间和精力，目前IoM庞大的数百个功能中，只有30%左右得到了充分的测试和打磨，绝大部分功能仍存在不少问题。

我们的目标是让IoM的CLI、Server、GUI、Implant都能达到CobaltStrike的稳定性、易用性和优雅度，并进一步拓展各种功能，实现覆盖全操作系统、全场景、全流程的Offensive基础设施。


## 目前的情况

### implant问题

1. implant的稳定性存在问题，包括各种操作系统版本兼容问题、BOF/PE兼容问题。部分情况会导致panic，部分存在内存泄漏，部分数据在server端处理不当。
2. implant有部分module缺乏实战验证，边界条件处理不完善。

### client/server

1. client的example和description存在大量错误，许多功能变动后文档未同步更新，导致CLI文档与实际用法不一致。
2. client的command部分功能失效，输出信息不符合预期，无法达到预期效果。
3. mals插件包中有大量插件未经充分测试，无法按预期工作。
4. 用户体验较差，TUI存在大量bug。
5. server部分代码可读性差，bug较多，亟需重构。
### wiki

文档是我们目前最头疼的问题。由于功能快速迭代，文档总是落后代码好几个版本，文档的表达和example可能已经过期，需要更新。

1. 急需文档校验、核对、优化。
2. 补全已实现功能但缺少文档的部分。
3. 文档表述存在歧义，难以理解，阅读逻辑过于抽象等问题。



## 期望达到的效果
### 入门级 - Issue Reporter

通过深度使用IoM发现问题：

- 提交bug报告（附带复现步骤）
- 提出功能需求和改进建议
- 反馈不合理的设计和低级bug
- 指出文档中的错误描述、歧义等

### 进阶级 - Contributor

协助解决具体问题（多数问题只需几行代码修改）：

1. 分析并定位问题
2. 编写修复代码（建议使用AI辅助）
3. 完成测试验证
4. 提交Pull Request

### 专家级 - Core Contributor

参与新功能开发和架构优化：

1. 发起需求并讨论技术方案
2. 实现完整功能模块（建议使用AI辅助）
3. 参与Code Review和迭代优化
4. 在IoM中留下你的技术印记

## 开发规范

### PR合并流程

**角色分配**

- 每个复杂功能分配一个Maintainer和至少一个Assignee
- Maintainer：功能实现者
- Assignee：代码审查和测试者

 **Review流程**

- Maintainer完成功能后通知Assignee进行review和测试
- 在PR中完成修改和讨论
- Maintainer与Assignee确认后联系 @m09ic 进行最终review

**文档要求**

- PR中附上测试截图和用法说明
- 说明测试环境和步骤
- 系统性功能PR通过后**立即**编写相关文档
- 新功能需添加对应的help信息
- 在wiki中补充使用文档
- 完成后通知 @m09ic
